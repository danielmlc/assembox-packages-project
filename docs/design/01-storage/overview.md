## 一、设计概述

### 1.1 职责

存储层为低代码平台提供配置数据和生成代码的持久化存储，支持：
- 配置元数据和内容的分离存储
- 草稿和发布版本的管理
- 配置的多层级继承（系统/全局/租户）
- 版本历史和快照管理
- 生成代码的备份存储

**层级说明**：

```
模块组 (Module Group)
  └── 模块 (Module)              ← 逻辑分组
       └── 版本 (Version)        ← 模块独立版本管理
            └── 组件 (Component) ← 最小配置单元
                 ├── 分类：服务端（Controller, Service, Entity, Repository, RPC等）
                 ├── 分类：前端（Form, Table, Filter, Export, Print, 路由等）
                 ├── 分类：模型（Entity, DTO等）
                 └── 配置类型：
                      ├── 运行时配置：不生成代码，运行时动态读取
                      └── 构建配置：生成代码，构建时使用
```

**关键概念说明**：

| 概念 | 说明 | 示例 |
|-----|------|------|
| 模块组 | 单个微服务，独立构建发布 | 订单服务、库存服务 |
| 模块 | 模块组内的逻辑分组 | 订单模块、支付模块 |
| 版本 | 模块的独立版本号 | v1.0.0, v1.1.0 |
| 组件 | 最小配置单元，按分类组织 | Entity, Service, Form, Table |

### 1.3 设计原则

1. **配置元数据分离**：数据库存元信息，OSS存配置内容（可配置）
2. **草稿和发布分离**：设计态和运行态隔离
3. **组件级继承**：以组件为最小粒度进行配置继承
4. **模块独立版本**：每个模块有自己的版本管理
5. **租户独立迭代**：租户定制版本与系统版本各自演进

## 二、技术选型

| 存储类型 | 技术 | 职责 | 备注 |
|---------|------|------|------|
| **数据库** | MySQL/TiDB | 配置元信息、关系、版本信息、状态 | 事务支持 |
| **对象存储** | OSS | 配置内容（JSON/YAML） | 组件可配置是否启用 |
| **版本控制** | Git | 原始配置文件的版本追溯 | 已发布配置 |
| **代码仓库** | Gitea | 生成代码的备份管理 | 构建产物 |
| **缓存** | Redis | 运行时配置缓存（可选） | 提升性能 |

### 3.2 数据流向

#### 3.2.1 草稿保存流程

```
┌──────────────┐
│   设计器      │ 用户编辑组件配置
└──────┬───────┘
       │ ① 保存草稿请求
       ▼
┌────────────────────────────────────────────────────────┐
│                     服务层                              │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ② 归档当前草稿到历史                               │ │
│  │                                                   │ │
│  │   IF (配置已有草稿):                              │ │
│  │     - 读取当前 draft_oss_key                      │ │
│  │     - 复制到 draft-history/ 路径                  │ │
│  │     - 记录到 ab_component_config_draft_history   │ │
│  └──────────────────────────────────────────────────┘ │
│                      │                                 │
│                      ▼                                 │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ③ 保存新草稿到 OSS                                 │ │
│  │                                                   │ │
│  │   - 生成 draft_oss_key 路径                       │ │
│  │   - 保存配置内容到 OSS draft/ 路径                │ │
│  └──────────────────────────────────────────────────┘ │
│                      │                                 │
│                      ▼                                 │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ④ 更新数据库元信息                                 │ │
│  │                                                   │ │
│  │   UPDATE ab_component_config SET                 │ │
│  │     draft_oss_key = ?,                           │ │
│  │     draft_version = draft_version + 1,           │ │
│  │     modifier_at = NOW()                          │ │
│  └──────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────┘
       │
       ▼ ⑤ 返回保存成功
┌──────────────┐
│   设计器      │
└──────────────┘
```

#### 3.2.2 发布流程（增量发布）

**发布粒度**：支持组件级、模块级、模块组级发布

```
┌──────────────┐
│   设计器      │ 用户选择发布范围
└──────┬───────┘
       │ ① 发布请求
       │    - 发布范围: 组件列表 / 模块列表 / 模块组
       │    - 租户: tenant_id (可选)
       ▼
┌────────────────────────────────────────────────────────┐
│                     服务层                              │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ② 确定待发布组件清单                               │ │
│  │                                                   │ │
│  │   IF (发布粒度 == 组件):                          │ │
│  │     待发布组件 = 选中的组件列表                    │ │
│  │   ELSE IF (发布粒度 == 模块):                     │ │
│  │     待发布组件 = 选中模块下的所有组件              │ │
│  │   ELSE IF (发布粒度 == 模块组):                   │ │
│  │     待发布组件 = 模块组下的所有组件                │ │
│  └──────────────────────────────────────────────────┘ │
│                      │                                 │
│                      ▼                                 │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ③ 批量发布组件 (仅更新选中的组件)                  │ │
│  │                                                   │ │
│  │   FOR EACH 待发布组件:                             │ │
│  │     - 验证草稿存在 (draft_oss_key != NULL)        │ │
│  │     - OSS: 复制 draft/ → published/               │ │
│  │     - UPDATE ab_component_config SET             │ │
│  │         status='published',                      │ │
│  │         publish_version = publish_version + 1,   │ │
│  │         published_oss_key = draft_oss_key,       │ │
│  │         published_at = NOW()                     │ │
│  │     - 记录 ab_config_history                      │ │
│  └──────────────────────────────────────────────────┘ │
│                      │                                 │
│                      ▼                                 │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ④ 推送配置到 Git                                   │ │
│  │                                                   │ │
│  │   - 导出本次发布的配置文件                         │ │
│  │   - git commit + push                            │ │
│  │   - git tag: publish/{timestamp}                 │ │
│  └──────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────┘
       │
       ▼ ⑤ 返回发布成功
┌──────────────┐
│   设计器      │ 发布完成,可手动触发构建
└──────────────┘
```

**关键说明**：
- 发布操作**不自动触发构建**
- **只更新被选中的组件**的 `publish_version`
- 未发布的组件保持原有版本号不变
- 发布后需要手动触发构建

#### 3.2.4 构建触发流程

**构建与发布分离**：构建由用户或 CI/CD 手动触发

```
┌──────────────┐
│ 用户/CI/CD    │ 触发构建
└──────┬───────┘
       │ ① 构建请求 (module_group, tenant)
       ▼
┌────────────────────────────────────────────────────────┐
│                     服务层                              │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ② 生成快照                                         │ │
│  │                                                   │ │
│  │   - 生成快照号 (S001, S002...)                    │ │
│  │   - 收集模块组下所有模块的组件清单                 │ │
│  │   - 记录每个组件的当前 publish_version            │ │
│  │   - 写入 ab_snapshot 表                           │ │
│  │                                                   │ │
│  │   快照记录构建时刻的完整配置状态:                  │ │
│  │   - 包含所有组件(无论是否最近发布)                │ │
│  │   - 锁定每个组件的 publish_version                │ │
│  └──────────────────────────────────────────────────┘ │
│                      │                                 │
│                      ▼                                 │
│  ┌──────────────────────────────────────────────────┐ │
│  │ ③ 触发构建任务                                     │ │
│  │                                                   │ │
│  │   - 传递 snapshot_id 给构建服务                    │ │
│  │   - 异步执行构建流程                               │ │
│  └──────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────┘
       │
       ▼ ④ 调用构建服务
┌────────────────────────────────────────────────────────┐
│                  构建服务                               │
│  - 根据 snapshot_id 读取配置 (见 3.2.5)               │
│  - 生成代码                                             │
│  - 推送到 Gitea                                         │
│  - 触发 CI/CD 部署                                      │
└────────────────────────────────────────────────────────┘
```

**关键说明**：
- **快照在构建时生成**,不是发布时
- 快照记录模块组的**完整配置状态**
- 支持基于快照的精确构建和回滚

## 四、核心设计

### 4.1 配置类型

| 配置类型 | 层级 | 是否生成代码 | 读取时机 | 示例 |
|---------|------|-------------|---------|------|
| **运行时配置** | 系统→全局→租户 | 否 | 运行时动态读取 | 权限规则、UI布局、字段显示规则 |
| **构建配置** | 系统|租户 | 是 | 构建时读取生成代码 | Entity、Service、Controller、Form |

**is_runtime 字段说明**：
- `is_runtime=1`：运行时配置，不生成代码
- `is_runtime=0`：构建配置，生成代码

### 4.2 配置继承

**组件级继承规则**：

```
运行时配置：租户 ← 全局 ← 系统
```

**继承查找逻辑**：
1. 查找租户层是否有该组件配置 → 有则使用
2. （运行时配置）查找全局层配置 → 有则使用
3. 使用系统层配置（必须存在）

**is_inheritable 字段控制**：
- `is_inheritable=1`：允许继承，支持租户/全局覆盖
- `is_inheritable=0`：不允许继承，如果取租户层就是租户层，取系统层就是系统层

### 4.3 版本管理

#### 4.3.1 模块版本

- 每个模块独立管理版本号（如 v1.0.0, v1.1.0）
- 业务块版本 = 各模块版本的组合
- 发布时指定业务块包含哪些模块及版本

#### 4.3.2 租户版本

- 每个产品（业务块）都有租户版本分支
- 租户定制基于某个系统版本创建租户分支
- 租户版本与系统版本独立迭代，互不影响

**版本分支示例**：
```
系统版本：
  订单模块 v1.0.0 → v1.1.0 → v1.2.0

租户 T001 版本（基于 v1.0.0 创建）：
  订单模块 v1.0.0-T001 → v1.0.1-T001 → v1.0.2-T001
  （独立迭代，不自动同步系统版本）
```

#### 4.3.3 草稿和发布版本

| 版本类型 | 字段 | OSS 路径 | 用途 |
|---------|------|---------|------|
| 草稿版本 | draft_version | draft/ | 设计器编辑 |
| 草稿历史 | - | draft-history/ | 版本回退 |
| 发布版本 | publish_version | published/ | 运行时/构建时 |
| 发布历史 | - | Git + DB | 审计追溯 |

### 4.4 快照机制

**快照生成时机**：
- 快照在**构建时生成**，不是发布时
- 每次触发构建时，生成新的快照号
- 快照记录构建时刻模块组的完整配置状态

**快照的作用**：
- 锁定构建时刻所有组件的发布版本
- 保证构建的配置一致性和可追溯性
- 支持基于快照的精确构建和回滚
- 支持多版本并存和灰度发布

**增量发布与快照**：
- 发布是增量的：只更新有变化的组件版本
- 快照是全量的：记录所有组件的当前版本
- 示例：
  - 第一次构建：组件 A(v1), B(v1), C(v1) → 快照 S001
  - 只发布组件 A：A 版本递增为 v2，B 和 C 保持 v1
  - 第二次构建：组件 A(v2), B(v1), C(v1) → 快照 S002

**快照清单结构**：
```json
{
  "snapshot_id": "S002",
  "module_group": "order-service",
  "version": "v1.0.0",
  "create_time": "2026-02-06T14:00:00Z",
  "modules": [
    {
      "module_code": "order",
      "version": "v1.0.0",
      "components": [
        {
          "type": "entity",
          "code": "order_main",
          "publish_version": 6,  // 最近发布过，版本递增
          "layer": "system"
        },
        {
          "type": "service",
          "code": "order_service",
          "publish_version": 3,  // 未发布，版本不变
          "layer": "system"
        },
        {
          "type": "controller",
          "code": "order_controller",
          "publish_version": 8,  // 最近发布过，版本递增
          "layer": "system"
        }
      ]
    }
  ]
}
```

### 4.5 配置内容存储策略

**配置信息 vs 配置内容**：

| 存储位置 | 存储内容 | 说明 |
|---------|---------|------|
| **数据库** | 配置元信息 | 模块、版本、组件、层级、状态、版本号 |
| **数据库** | 配置内容（小） | 如果配置内容较小，直接存 DB |
| **OSS** | 配置内容（大） | 如果开启 OSS 存储，内容存 OSS |

**is_store_oss 字段控制**：
- `is_store_oss=1`：配置内容存 OSS（draft_oss_key, published_oss_key）
- `is_store_oss=0`：配置内容存 DB（draft_content, published_content）

**OSS 路径规范**：
```
draft/{module}/{version}/{layer}/{component_type}/{component_code}.json
draft-history/{module}/{version}/{layer}/{component_type}/{component_code}/{timestamp}.json
published/{module}/{version}/{layer}/{component_type}/{component_code}/{publish_version}.json
```

## 五、核心表结构概要

> 详细表结构设计见后续文档，此处仅列出核心表及其职责。

### 5.1 核心表列表

| 表名 | 职责 | 关键字段 |
|-----|------|---------|
| ab_module_group | 模块组（Module Group）定义 | block_code, name, description |
| ab_module | 模块定义 | module_code, group_id, name |
| ab_version | 版本管理 | module_id, version, tenant_id, status |
| ab_component | 组件类型注册表 | type_code, is_runtime, is_inheritable, is_store_oss |
| ab_component_config | 组件配置索引 | module_id, version_id, type, code, layer, status, draft_version, publish_version, draft_oss_key, published_oss_key |
| ab_component_config_draft_history | 组件配置草稿历史 | config_id, draft_version, oss_key, created_at |
| ab_snapshot | 快照管理 | snapshot_code, manifest, module_group_id, version |
| ab_config_history | 发布历史 | config_id, publish_version, oss_key, published_at |

### 5.2 表关系说明

```
ab_module_group (模块组)
  └── ab_module (模块)
       └── ab_version (版本)
            └── ab_component_config (组件配置)
                 ├── ab_component_config_draft_history (草稿历史)
                 └── ab_config_history (发布历史)

ab_component (组件类型注册表)
  └── ab_component_config (引用 type_code)

ab_snapshot (快照)
  └── 关联 ab_module_group
       └── manifest 中记录各模块组件的版本
```

### 5.3 关键表说明

**ab_component_config**（组件配置索引表）：
- 存储配置的元信息和状态
- `draft_version`：草稿版本号，每次保存草稿时递增
- `publish_version`：发布版本号，每次发布时递增
- `draft_oss_key`：草稿配置在 OSS 的路径
- `published_oss_key`：已发布配置在 OSS 的路径
- `status`：配置状态（draft/published）

**ab_component_config_draft_history**（草稿历史表）：
- 记录草稿的每次保存历史
- 支持草稿版本回退
- `oss_key`：指向 OSS draft-history/ 路径
- 每次保存草稿前，当前草稿归档到此表

**ab_config_history**（发布历史表）：
- 记录配置的每次发布历史
- 支持发布回滚和审计追溯
- `oss_key`：指向 OSS published/ 路径
- 每次发布时记录

## 七、设计参考

本设计参考了之前的存储层设计（[docs/design/01-storage](../01-storage)），主要优化点：

1. **简化概念层级**：去掉流水线概念，简化为业务块→模块→组件
2. **明确配置分类**：运行时配置 vs 构建配置，替代 is_runtime 单一标记
3. **租户版本独立**：租户定制版本独立迭代，不自动同步系统版本
4. **存储策略可配**：组件级控制是否存储到 OSS
5. **保留核心优点**：草稿/发布分离、快照机制、配置继承、组件级管理
